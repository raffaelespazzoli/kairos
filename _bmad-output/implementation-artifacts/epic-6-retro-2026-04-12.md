## DORA Query Optimization (Retro Deep-Dive Finding)

### Problem

Each DORA metric makes 3 sequential HTTP calls to Prometheus:
1. Instant query for current value
2. Instant query with `offset` for previous period value
3. Range query for time series chart data

At single-app scale (Epic 6): 12 calls, manageable.
At team-dashboard scale (Epic 7, 10 apps): 120 calls, unacceptable.

### Analysis

- **Call 1 is redundant:** The range query (Call 3) already returns the full time series. The last data point IS the current value.
- **Call 2 cannot be eliminated by extending range:** Extending the range to 2× and splitting produces incorrect trend comparisons when data is sparse (e.g., app with 20 days of data requested over 60 days → partial overlap skews the comparison). Prometheus `offset` handles this correctly.
- **Calls 2 and 3 are independent:** They can be parallelized within each metric.

### Solution: Option A+

- Drop Call 1 (redundant instant query) — extract current value from range query's last data point
- Keep Call 2 (offset instant query) — Prometheus handles windowing correctly
- Parallelize Call 2 and Call 3 within each metric via CompletableFuture

| Metric | Before | After |
|--------|--------|-------|
| Calls per metric | 3 serial | 2 parallel |
| Total calls (1 app) | 12 | 8 |
| Total calls (10 apps) | 120 | 80 |
| Effective latency | 3 round trips | **1 round trip** |

### Status

Applied immediately after retrospective as prep for Epic 7.
