# Story 3.2: DevSpaces Launch & Vault Secret Navigation

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a developer,
I want to launch a DevSpaces web IDE for my application and navigate to my application's secrets in Vault directly from the portal,
So that I can start coding or manage secrets without manually finding the right URL.

## Acceptance Criteria

1. **DevSpaces deep link on ApplicationOverviewPage**
   - **Given** a developer is viewing the ApplicationOverviewPage for an onboarded application
   - **When** the page renders
   - **Then** an "Open in DevSpaces ↗" DeepLinkButton is displayed in the application header area
   - **And** the link is generated by DeepLinkService using the application's Git repository URL
   - **And** clicking it opens DevSpaces in a new tab scoped to the application's repository

2. **DevSpaces not configured graceful handling**
   - **Given** DevSpaces is not configured (`portal.devspaces.url` is empty)
   - **When** the ApplicationOverviewPage renders
   - **Then** the "Open in DevSpaces ↗" button is not shown

3. **Vault secrets management section on Settings tab**
   - **Given** a developer navigates to the Application Settings tab
   - **When** the ApplicationSettingsPage renders
   - **Then** a "Secrets Management" section is displayed
   - **And** for each environment in the application's promotion chain, a Vault deep link is shown
   - **And** each link is labeled "Manage secrets in Vault ↗" with the environment name (e.g., "dev — Manage secrets in Vault ↗")
   - **And** each link is scoped to the Vault path: `/applications/<team>/<team>-<app>-<env>/static-secrets`

4. **Vault URL not configured graceful handling**
   - **Given** the Vault URL is not configured
   - **When** the Application Settings page renders
   - **Then** the Secrets Management section shows a message: "Vault URL not configured — contact your platform administrator"
   - **And** no deep link buttons are shown

5. **Onboarding wizard completion step DevSpaces link**
   - **Given** the onboarding wizard completion step (Story 2.5)
   - **When** the "Open in DevSpaces ↗" placeholder link was shown
   - **Then** it is now functional, using DeepLinkService to generate the correct URL

### DeepLinkConfig — Configuration Mapping

```java
package com.portal.deeplink;

import io.smallrye.config.ConfigMapping;
import io.smallrye.config.WithDefault;
import java.util.Optional;

@ConfigMapping(prefix = "portal")
public interface DeepLinkConfig {

    Optional<String> devspacesUrl();

    Optional<String> vaultUrl();

    Optional<String> argocdUrl();

    Optional<String> grafanaUrl();

    Optional<String> grafanaDashboardId();

    TektonConfig tekton();

    interface TektonConfig {
        Optional<String> dashboardUrl();
    }
}
```

Use `Optional<String>` for all URLs — empty/missing means "not configured" and deep links should return null. Do NOT use `@WithDefault` with empty strings — use `Optional` to cleanly distinguish "unconfigured" from "configured with empty string".

**Important:** The config prefix is `portal` (not `portal.deeplink`) because the properties are already defined as `portal.devspaces.url`, `portal.vault.url`, etc. in the architecture. Use `@ConfigMapping(prefix = "portal")` and let SmallRye map the nested keys. Quarkus converts `portal.devspaces.url` → `devspacesUrl()` accessor via kebab-case-to-camelCase mapping. However, verify that these don't conflict with existing `@ConfigProperty` usages of `portal.*` keys (e.g., `portal.argocd.url` is used by the ArgoCD adapter via `@ConfigProperty`). If SmallRye `@ConfigMapping` conflicts with `@ConfigProperty` on the same keys, use `@ConfigProperty` instead:

```java
@ApplicationScoped
public class DeepLinkService {

    @ConfigProperty(name = "portal.devspaces.url")
    Optional<String> devspacesUrl;

    @ConfigProperty(name = "portal.vault.url")
    Optional<String> vaultUrl;

    // ... other URLs
}
```

This pattern is safer for a project that already uses `@ConfigProperty` for some `portal.*` keys.

### DeepLinkService — URL Generation

```java
package com.portal.deeplink;

import jakarta.enterprise.context.ApplicationScoped;
import org.eclipse.microprofile.config.inject.ConfigProperty;
import java.util.Optional;

@ApplicationScoped
public class DeepLinkService {

    @ConfigProperty(name = "portal.devspaces.url")
    Optional<String> devspacesUrl;

    @ConfigProperty(name = "portal.vault.url")
    Optional<String> vaultUrl;

    public String generateDevSpacesLink(String gitRepoUrl) {
        return devspacesUrl
                .filter(url -> !url.isBlank())
                .map(url -> url + "/#/" + gitRepoUrl)
                .orElse(null);
    }

    public String generateVaultLink(String team, String app, String env) {
        return vaultUrl
                .filter(url -> !url.isBlank())
                .map(url -> url + "/ui/vault/secrets/applications/"
                        + team + "/" + team + "-" + app + "-" + env + "/static-secrets")
                .orElse(null);
    }
}
```

**Key design decisions:**
- Returns `null` when the tool's base URL is not configured — the frontend interprets null as "don't render the button"
- No URL encoding needed for DevSpaces — Git repo URLs are already URL-safe
- Vault path follows the architecture-defined convention: `/applications/{team}/{team}-{app}-{env}/static-secrets`
- The Vault deep link path uses the **team name** (from the `Team` entity), **application name** (from the `Application` entity), and **environment name** (from the `Environment` entity, e.g., "dev", "qa", "prod")

### Configuration Properties

Add to `application.properties`:

```properties
# Deep Links — DevSpaces
portal.devspaces.url=${DEVSPACES_URL:}

# Deep Links — Vault UI (for secret navigation — distinct from portal.secrets.vault.url which is the API endpoint)
portal.vault.url=${VAULT_UI_URL:}
```

**Critical distinction:** `portal.vault.url` is for the Vault **UI** (user-facing deep links to the web console). `portal.secrets.vault.url` is for the Vault **API** (backend credential retrieval). These may differ in on-prem deployments (e.g., API at `https://vault.internal:8200`, UI at `https://vault.company.com`).

Add to `application-dev.properties` (or dev profile in main):

```properties
%dev.portal.devspaces.url=https://devspaces.example.com
%dev.portal.vault.url=https://vault.example.com
```

### Update EnvironmentMapper and EnvironmentService

`EnvironmentMapper.merge()` needs additional parameters for vault deep links. The service must:
1. Look up the team name from the application's team
2. Call `DeepLinkService.generateVaultLink(teamName, appName, envName)` for each environment
3. Pass the vault deep links to the mapper

```java
// In EnvironmentService:
@Inject
DeepLinkService deepLinkService;

public EnvironmentChainResponse getEnvironmentChain(Long teamId, Long appId) {
    Application app = Application.findById(appId);
    if (app == null || !app.teamId.equals(teamId)) {
        throw new NotFoundException();
    }

    Team team = Team.findById(teamId);
    String teamName = team != null ? team.name : "";

    List<Environment> environments =
            Environment.findByApplicationOrderByPromotionOrder(appId);

    Map<Long, String> clusterNames = resolveClusterNames(environments);

    // Generate vault deep links per environment
    Map<String, String> vaultLinks = environments.stream()
            .collect(Collectors.toMap(
                    e -> e.name,
                    e -> {
                        String link = deepLinkService.generateVaultLink(teamName, app.name, e.name);
                        return link != null ? link : "";
                    }));

    String argocdError = null;
    List<EnvironmentStatusDto> statuses = List.of();
    try {
        statuses = argoCdAdapter.getEnvironmentStatuses(app.name, environments);
    } catch (PortalIntegrationException e) {
        argocdError = e.getMessage();
    }

    List<EnvironmentChainEntryDto> entries =
            EnvironmentMapper.merge(environments, statuses, clusterNames, vaultLinks);

    return new EnvironmentChainResponse(entries, argocdError);
}
```

Update `EnvironmentMapper.merge()` to accept and include vault links:

```java
public static List<EnvironmentChainEntryDto> merge(
        List<Environment> environments,
        List<EnvironmentStatusDto> statuses,
        Map<Long, String> clusterNames,
        Map<String, String> vaultLinks) {

    Map<String, EnvironmentStatusDto> statusByEnv = statuses.stream()
            .collect(Collectors.toMap(
                    EnvironmentStatusDto::environmentName, s -> s));

    return environments.stream()
            .map(env -> {
                EnvironmentStatusDto status = statusByEnv.get(env.name);
                String vaultLink = vaultLinks.getOrDefault(env.name, null);
                return new EnvironmentChainEntryDto(
                        env.name,
                        clusterNames.getOrDefault(env.clusterId, null),
                        env.namespace,
                        env.promotionOrder,
                        status != null ? status.status().name() : "UNKNOWN",
                        status != null ? status.deployedVersion() : null,
                        status != null ? status.lastDeployedAt() : null,
                        status != null ? status.argocdDeepLink() : null,
                        vaultLink != null && !vaultLink.isEmpty() ? vaultLink : null);
            })
            .toList();
}
```

**Important UX detail:** The AC specifies the label format as "dev — Manage secrets in Vault ↗" with the environment name. The existing `DeepLinkButton` component has a hardcoded label format "Open in {toolName} ↗". For the Settings page, the label needs customization. Two options:

**Option A (recommended):** Add an optional `label` prop to `DeepLinkButton`:

```typescript
interface DeepLinkButtonProps {
  href: string;
  toolName: string;
  label?: string;
}

export function DeepLinkButton({ href, toolName, label }: DeepLinkButtonProps) {
  return (
    <Button
      variant="link"
      component="a"
      href={href}
      target="_blank"
      rel="noopener noreferrer"
      isInline
    >
      {label ?? `Open in ${toolName} ↗`}

  );

```

Then render: `<DeepLinkButton href={env.vaultDeepLink} toolName="Vault" label={`${env.environmentName} — Manage secrets in Vault ↗`} />`

**Option B:** Render the environment name outside the DeepLinkButton in a DescriptionList term, and keep the button text generic. This is cleaner if the Description List layout already communicates which environment.

Go with Option A for flexibility, but render the environment name in the `DescriptionListTerm` and use the generic button label for consistency with other deep links. The AC says `"dev — Manage secrets in Vault ↗"` as the label format, so use the custom label prop to include the environment name prefix.

### Anti-Patterns to Avoid

- **DO NOT** generate deep links on the frontend — all URL construction happens in backend `DeepLinkService`
- **DO NOT** hardcode tool URLs — always read from configuration
- **DO NOT** show broken deep link buttons — if URL is null, don't render the button
- **DO NOT** confuse `portal.vault.url` (UI deep links) with `portal.secrets.vault.url` (API credential retrieval)
- **DO NOT** import `DeepLinkService` directly in REST resources — inject via CDI `@Inject`
- **DO NOT** use `any` types in TypeScript — all interfaces are strongly typed
- **DO NOT** call `fetch()` directly — use `apiFetch()` wrapper
- **DO NOT** use absolute URLs in frontend API calls — all paths are relative
- **DO NOT** use `--pf-v5--` CSS tokens — use `--pf-t--global--` (PF6)
- **DO NOT** create custom CSS for layouts PatternFly provides — use PF6 DescriptionList, Card, Alert
- **DO NOT** return 403 for cross-team access — return 404
- **DO NOT** cache deep link URLs — they derive from configuration which is static at runtime

### What NOT to Build

- **No ArgoCD deep links on environment cards** — that's Story 3.3
- **No Grafana deep links** — that's Epic 6
- **No Tekton deep links** — that's Epic 4
- **No secret CRUD** — the portal only links to Vault's native UI; no secret read/write through the portal
- **No DevSpaces API integration** — DevSpaces is deep-link-only, no API calls
- **No Vault API calls for secrets management** — only URL generation for the Vault UI
- **No auto-refresh** on settings page — manual navigation only
