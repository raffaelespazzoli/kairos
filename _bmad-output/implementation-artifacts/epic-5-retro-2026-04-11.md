## New Rule — Interactive Control Rendering (Added to project-context.md)

### Problem

Frontend UX state management findings accounted for 6 of 14 review findings in Epic 5. Components with multi-dimensional conditional rendering (environment status × role × loading state × data availability) had recurring edge-case failures: silent no-ops, vanishing controls, mismatched labels, and hidden errors.

### Solution

New rule added to project-context.md with 6 sub-rules:

1. **Guard on all required inputs before rendering** — never render an interactive element that will silently no-op on click
2. **Preserve control presence during loading** — disable + Spinner, never unmount
3. **Match control labels to the actual action** — deploy vs promote, not the underlying API
4. **Surface upstream data failures inline** — explain why actions are unavailable
5. **Document the state matrix** — comment-level truth table for 3+ state dimensions
6. **Test every guard rule** — each guard condition must have a corresponding frontend test

### Enforcement

Same as `require*`: if a reviewer sees a guard in JSX with no matching test, it's a review rejection.

## Design Decisions Captured During Retro

### PromQL as Configuration (for Epic 6)

Prometheus queries externalized as configuration under `portal.prometheus.queries.*`. The PrometheusAdapter is a generic query executor that substitutes `{namespace}` and `{interval}` placeholders at runtime. Benefits:
- Platform teams tune queries without code changes
- Dev agents don't need PromQL expertise to implement stories
- Different Prometheus setups with different metric names handled via config
- DORA metric queries follow the same pattern

Added to project-context.md during retro.
