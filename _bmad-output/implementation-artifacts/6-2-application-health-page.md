# Story 6.2: Application Health Page

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a developer,
I want a health page showing golden signal metrics per environment and direct links to Grafana,
so that I can verify my application is running well and investigate further when it's not.

## Acceptance Criteria

1. **Health tab renders a section per environment ordered by promotion chain**
   - **Given** a developer navigates to the Health tab for an application
   - **When** the `ApplicationHealthPage` renders
   - **Then** it calls `GET /api/v1/teams/{teamId}/applications/{appId}/health`
   - **And** displays a section for each environment in the promotion chain (ordered by `promotion_order`)
   - **And** each section shows the environment name, overall health status badge, and golden signal metrics

2. **GoldenSignalsPanel displays four metric cards with color coding**
   - **Given** the `GoldenSignalsPanel` component renders for an environment
   - **When** health data is available
   - **Then** it displays four metric cards in a row:
     - Latency: p95 value with unit (e.g., "245ms")
     - Traffic: request rate (e.g., "42 req/s")
     - Errors: error percentage (e.g., "0.3%")
     - Saturation: CPU/memory (e.g., "CPU 45%, Mem 62%")
   - **And** each metric card uses color coding: green for healthy range, yellow for warning range, red for critical range

3. **Unhealthy/Degraded environments are visually highlighted**
   - **Given** an environment has status "Unhealthy" or "Degraded"
   - **When** the health section renders
   - **Then** the offending metrics are highlighted in red or yellow
   - **And** the overall status badge shows "✕ Unhealthy" (danger) or "⟳ Degraded" (warning)

4. **Grafana deep link per environment health section**
   - **Given** the Grafana deep link is configured
   - **When** each environment's health section renders
   - **Then** a "View in Grafana ↗" `DeepLinkButton` is displayed
   - **And** the link uses the `grafanaDeepLink` from the health API response (generated by `DeepLinkService.generateGrafanaLink()`)
   - **And** clicking opens the Grafana dashboard in a new tab pre-filtered to that namespace

5. **Environment chain cards enriched with Prometheus health + Grafana links**
   - **Given** the environment chain cards (from Epic 2)
   - **When** health data is available from this epic
   - **Then** the environment chain card health badges now reflect live golden signal data from Prometheus (in addition to the ArgoCD sync/health status)
   - **And** the Grafana deep link on expanded `EnvironmentCard` is now functional (currently disabled placeholder)
   - **And** uses `entry.grafanaDeepLink` already present in `EnvironmentChainEntry`

6. **Prometheus unreachable — graceful per-environment degradation**
   - **Given** Prometheus is unreachable for a specific environment
   - **When** the health page renders
   - **Then** that environment's section shows "Health data unavailable" in grey
   - **And** a PatternFly inline `Alert` (warning) identifies the affected system
   - **And** other environments with available data render normally

7. **No traffic — "No Data" empty state**
   - **Given** no traffic has reached a newly deployed application
   - **When** the health page renders
   - **Then** the environment section shows "No Data" with a message: "Metrics will appear once the application receives traffic"

8. **Resource ownership validation (mandatory scoping pattern)**
   - **Given** a request targets a resource outside the caller's team scope
   - **Then** 404 is returned (handled by backend — frontend shows `ErrorAlert`)
   - **Given** a non-existent application
   - **Then** 404 is returned (frontend shows `ErrorAlert`)

### Backend API Response Shape

The health endpoint returns this structure (created in Story 6.1):

```typescript
// GET /api/v1/teams/{teamId}/applications/{appId}/health
interface HealthResponse {
  environments: EnvironmentHealthDto[];
}

interface EnvironmentHealthDto {
  environmentName: string;
  healthStatus: HealthStatusDto | null;  // null when error occurred
  grafanaDeepLink: string | null;
  error: string | null;                  // populated when Prometheus failed for this env
}

interface HealthStatusDto {
  status: 'HEALTHY' | 'UNHEALTHY' | 'DEGRADED' | 'NO_DATA';
  goldenSignals: GoldenSignal[];
  namespace: string;
}

interface GoldenSignal {
  name: string;
  value: number;
  unit: string;
  type: 'LATENCY_P50' | 'LATENCY_P95' | 'LATENCY_P99' | 'TRAFFIC_RATE' | 'ERROR_RATE' | 'SATURATION_CPU' | 'SATURATION_MEMORY';
}
```

**Critical:** The `value` for latency signals is in SECONDS (e.g., `0.245` = 245ms). The frontend must convert to milliseconds for display.

**Critical:** The `value` for saturation signals is a percentage (e.g., `45.0` = 45%). Error rate is also a percentage (e.g., `0.3` = 0.3%).

### Environment Chain Health Enrichment (Task 7b)

The environment chain cards currently display health status from ArgoCD (`EnvironmentChainEntry.status`). AC #5 requires that chain badges also reflect Prometheus golden signal data. The approach is **additive and non-blocking**:

1. `ApplicationOverviewPage` calls BOTH `useEnvironments()` AND `useHealth()` in parallel
2. The chain renders immediately from environment data (ArgoCD status)
3. When health data arrives, it enriches the chain card display
4. Health fetch failure is silent — chain works fine with ArgoCD-only data

**Status merging logic:** When Prometheus and ArgoCD disagree, show the MORE SEVERE status:

| ArgoCD Status | Prometheus Status | Displayed Status |
|---------------|-------------------|-----------------|
| HEALTHY | HEALTHY | HEALTHY (green) |
| HEALTHY | DEGRADED | DEGRADED (yellow) |
| HEALTHY | UNHEALTHY | UNHEALTHY (red) |
| UNHEALTHY | * | UNHEALTHY (red) |
| DEPLOYING | * | DEPLOYING (yellow, no override) |
| NOT_DEPLOYED | * | NOT_DEPLOYED (grey, no override) |
| UNKNOWN | HEALTHY | HEALTHY (Prometheus overrides unknown) |
| * | NO_DATA | Keep ArgoCD status (Prometheus has nothing) |

Pass health info through: `EnvironmentChain` → `EnvironmentCard`. Keep the `EnvironmentCard` props backward-compatible — `healthInfo` is optional and defaults to undefined.

### Metric Value Formatting

Implement a `formatMetricValue` utility function (can be in `GoldenSignalsPanel.tsx` or a shared util):

```typescript
function formatMetricValue(signal: GoldenSignal): string {
  switch (signal.type) {
    case 'LATENCY_P50':
    case 'LATENCY_P95':
    case 'LATENCY_P99':
      // Backend returns seconds, display as milliseconds
      return signal.value >= 1 
        ? `${signal.value.toFixed(2)}s` 
        : `${Math.round(signal.value * 1000)}ms`;
    case 'TRAFFIC_RATE':
      return `${signal.value.toFixed(1)} req/s`;
    case 'ERROR_RATE':
      return `${signal.value.toFixed(1)}%`;
    case 'SATURATION_CPU':
    case 'SATURATION_MEMORY':
      return `${Math.round(signal.value)}%`;
    default:
      return `${signal.value}`;
  }
}
```

### Metric Card Color Logic

Each metric card's color is determined by comparing the value against thresholds. These are frontend-only display thresholds (the backend derives overall health status using its own configurable thresholds):

| Metric | Green | Yellow | Red |
|--------|-------|--------|-----|
| Latency (p95) | <500ms (<0.5s) | 500ms-1000ms | >1000ms (>1s) |
| Error Rate | <1% | 1%-5% | >5% |
| Saturation (CPU or Mem) | <70% | 70%-90% | >90% |
| Traffic | Always blue/default | — | — |

Map to CSS tokens:
- Green: `--pf-t--global--color--status--success--default`
- Yellow: `--pf-t--global--color--status--warning--default`
- Red: `--pf-t--global--color--status--danger--default`
- Blue/default: `--pf-t--global--color--brand--default`

### GoldenSignalsPanel Layout

Display four cards: Latency, Traffic, Errors, Saturation. Extract the relevant signals from the `goldenSignals` array by `type`:

- **Latency card:** Show p95 as the primary value. Optionally show p50 and p99 as secondary text below.
- **Traffic card:** Show `TRAFFIC_RATE` value.
- **Errors card:** Show `ERROR_RATE` value.
- **Saturation card:** Show both `SATURATION_CPU` and `SATURATION_MEMORY` — e.g., "CPU 45%, Mem 62%". Use the HIGHER of the two values to determine the card's color threshold.
