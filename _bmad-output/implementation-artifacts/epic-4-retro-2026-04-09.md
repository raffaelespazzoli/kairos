## Successes

- **100% story completion** with zero test regressions — the pre-flight test gate from Epic 3's retro was applied successfully across all 5 stories
- **168 new tests** (101 backend, 67 frontend) — test count grew from ~500 to ~668, providing a strong safety net

- **Code reviews caught all 11 real issues** — including 3 authorization scoping gaps that could have been security problems
- **Cross-cutting changes executed cleanly** — GitProvider was extended twice (4.4 and 4.5), touching all 5 implementations each time, without regressions
- **First Kubernetes API integration** was clean — TektonAdapter pattern (programmatic client per call, try-with-resources, Vault-issued credentials) established a reusable template for future K8s interactions

## Challenges

- **Authorization scoping gaps recurred** — Stories 4.1 and 4.2 both missed resource ownership checks (team→app, app→build) that code reviews caught. The pattern was ad-hoc — each service wrote its own ownership validation
- **DTO evolution caused cascading updates** — `BuildSummaryDto` was extended 3 times across the epic (original in 4.1, +completedAt/duration in 4.2, +imageReference in 4.4). `BuildDetailDto` was extended in 4.4 (+commitSha). Each extension required updating all callers and tests
- **Fabric8/Tekton API surface surprises** — Story 4.1 discovered classes live in `io.fabric8.tekton.v1` (not `.pipeline.v1`); Story 4.2 discovered Tekton uses `io.fabric8.knative.pkg.apis.Condition` (not K8s Condition); `PipelineRunList.getItems()` returns immutable lists
- **PermissionFilter heuristic needed patching** — Story 4.2 required adding "logs" to `ACTION_SEGMENTS` because the `isId()` heuristic couldn't distinguish `/builds/{buildId}/logs` from `/builds/{buildId}/{subresource}`
- **Story 4.3 had the most review findings (4)** — the only story using claude-4.6-opus instead of opus-high-thinking. Findings were about implementation precision vs AC requirements (inline insert vs full refresh, AbortSignal not wired, no retry path)

## Key Insights

1. **Authorization scoping is an architectural concern, not a developer memory test** — recurring gaps across stories proved that relying on developers to remember ownership checks doesn't scale. Formalized as the `require*` pattern in project-context.md
2. **DTO evolution across sequential stories creates compounding change** — when a DTO is defined in story N and extended in stories N+1 and N+2, each extension touches an expanding set of callers. Future story specs should anticipate downstream DTO needs
3. **Code reviews remain the critical quality gate** — 11 findings in 5 stories, with 3 security-relevant. The review process works, but shifting checks left (via documented patterns and AC templates) reduces review burden
4. **Model selection may correlate with review finding count** — Story 4.3 (opus, 4 findings) vs Stories 4.1/4.2/4.4/4.5 (opus-high-thinking, 0-2 findings each). Small sample, but worth monitoring in Epic 5
5. **Cross-cutting interface changes are manageable with the provider pattern** — adding methods to GitProvider's 5 implementations was clean both times. The factory pattern + dev mock pattern keeps this tractable

## Authorization Pattern — Formalized During Retro

### Problem

Authorization scoping gaps (team→app, app→build ownership checks) were caught by code reviews in 3 out of 5 stories. The pattern was ad-hoc — each service implemented its own checks.

### Solution (Three-Layer Defense)

Proposed by Winston (Architect), Amelia (Developer), and Mary (Analyst) during the retro:

| Layer | What | Effect |
|---|---|---|
| **Requirements** | Standard Authorization AC template in every endpoint story | Makes scoping requirements explicit, not implicit |
| **Architecture** | `require*` named methods for each ownership edge | Makes omission visible and auditable during review |
| **Documentation** | Pattern documented in project-context.md | Provides AI agents and reviewers with explicit rules |

### Codebase Scan Results

Scanned all 8 `*Service.java` files. Found 3 deviations, all resolved during retro:

| # | Service | Deviation | Resolution |
|---|---|---|---|
| 1 | `ApplicationService.getApplicationsForTeam` | No service-level team validation | ✅ Fixed — added TeamContext check + 2 new tests |
| 2 | `OnboardingService` | Uses TeamContext instead of path teamId | ✅ Documented as accepted variation (app doesn't exist yet) |
| 3 | `ClusterService` | No team scoping | ✅ Documented as accepted exception (admin-only, global resources) |

All services that call external platform adapters (TektonAdapter, ArgoCdAdapter, GitProvider, RegistryAdapter) correctly validate ownership before making adapter calls.
